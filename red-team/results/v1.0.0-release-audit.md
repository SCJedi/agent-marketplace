## Critical Issues Found and Fixed

### 1. Content hash not using unicode normalization (SECURITY FIX)
**File:** `src/db.js` line 746
**Issue:** `contentPublishWithHash()` computed SHA-256 hash directly from raw `content_text` without calling the existing `normalizeForHash()` function. This meant zero-width characters, RTL overrides, combining diacriticals, and fullwidth characters would produce different hashes, allowing subtle content manipulation to bypass divergence detection.
**Fix:** Changed to normalize text via `normalizeForHash()` before hashing. Now zero-width chars, RTL overrides, soft hyphens, combining marks, and unicode spaces are stripped before hash computation.
**Impact:** High -- this was the core defense against unicode-based content integrity attacks.

### 2. Missing DB migrations for nodes table columns (CRASH FIX)
**File:** `src/db.js` migrations section
**Issue:** The `nodes` table schema includes `deposit`, `probation_remaining`, `publish_count`, `flagged`, and `flag_reason` columns in the CREATE TABLE, but older databases created before these columns were added would crash with "no such column: flagged" when the publish endpoint called `nodeIsFlagged()`.
**Fix:** Added ALTER TABLE migrations for all five columns, matching the existing migration pattern.
**Impact:** Medium -- would crash the publish endpoint for any node running an older DB.

### 3. Trending endpoint only accepted day format (CONSISTENCY FIX)
**File:** `src/routes/market.js`
**Issue:** PROTOCOL.md documents `1h`, `24h` as valid trending periods, but the server only accepted `Nd` format (e.g., `7d`, `30d`). Requests with `24h` returned a 400 error.
**Fix:** Added hour-format parsing (`Nh`) that converts to fractional days for the DB query.
**Impact:** Low -- API inconsistency with protocol documentation.

### 4. Red team test RT-A005 updated for normalization
**File:** `red-team/scenarios/api-security/RT-A005-hash-collision.js`
**Issue:** The test expected ALL unicode variants to produce different hashes. With normalization, some variants (zero-width chars, whitespace) correctly produce the SAME hash as the original, which is the desired security behavior (attack neutralized).
**Fix:** Updated test logic to recognize both outcomes as valid: hash matches original (normalization stripped attack) OR hash differs (divergence detected and flagged).

## Security Posture Assessment

**SQL Injection:** All queries use parameterized statements. DEFENDED.

**XSS:** JSON-only API, no HTML rendering in API responses. Dashboard serves static HTML. DEFENDED.

**Auth Bypass:** API key validation on protected routes. Anonymous access allowed for public content (by design). Owner-only whitelist management. DEFENDED.

**Path Traversal:** No file system access from API inputs. Content stored in SQLite, not filesystem. NOT APPLICABLE.

**Content Integrity:** Unicode normalization (NFKD + zero-width/RTL/combining mark stripping) on hash computation. Multi-provider consensus detection. Content flagging for divergent hashes. DEFENDED.

**Identity Spoofing:** API key required when claiming a registered node identity. Body-claimed provider_id overridden by authenticated identity. DEFENDED.

**Sybil Attacks:** Domain deduplication in verifier selection. Registration rate limiting. Deposit requirements. Publisher probation. DEFENDED.

**Rate Limiting:** Per-provider publish rate limits. Per-endpoint registration rate limits. Fastify global rate limiting. DEFENDED.

**Information Leakage:** API keys hashed (SHA-256, truncated first4...last4) in transaction ledger. No raw keys in responses except at registration time. ACCEPTABLE.

## Known Limitations (Honest Assessment)

1. **No real payment system.** All transactions are recorded at price 0 with "free" payment method. The ledger and reputation work, but no actual value transfer happens.

2. **CORS is fully open** (`origin: true`). Acceptable for a P2P protocol where any node talks to any other, but should be configurable for production deployments.

3. **Search uses LIKE queries.** Works for moderate data sizes but won't scale to millions of records. Full-text search (FTS5) would be needed for production scale.

4. **Economic red team scenarios not in CI runner.** The 9 economic and adversarial scenarios exist as standalone files but aren't registered in the CI runner's suite registry. They can be run individually.

5. **No TLS enforcement.** The protocol spec mentions HTTPS but the implementation doesn't enforce it. Peer-to-peer communication over HTTP is fine for local development but needs TLS for production.

6. **Seed nodes are localhost only.** Production deployment would need real seed node endpoints.

7. **No content size limits enforced.** The protocol mentions `max_content_size_bytes` in node advertisements but the server doesn't enforce maximum content sizes on publish.
